## surfaces
The current product is **multi‑surface**, spanning the full purchase journey. Each Upsell.com "Page" corresponds to a surface; the first shipped Page in the new Flow Builder is **Checkout**, with product/cart/thank‑you/order‑status added via app blocks/extensions. Mapped to our internal extension‑type vocabulary:

- **checkout.upsell** (confirmed) — In‑checkout upsell widgets (Shopify Plus for true checkout upsell). Shows product upsell / recommendations / banner widgets in checkout via checkout UI extensions. First‑class "Checkout" Page in Flow Builder.
- **postPurchase.offer** (confirmed) — Core historical surface: **post‑purchase page** (one‑click upsell after payment, before order confirmation) and the **Thank‑You / Order‑Status page**. Renders product upsell, bundle, recommendations, timer, banner, birthday collector, order summary, custom forms. Uses Shopify post‑purchase checkout extensions; subject to Shopify's post‑purchase gateway/vaulted‑payment limitations (has a dedicated "supported payment processors" article).
- **checkout.block** (confirmed) — Announcement Bar, Banner, Timer, Order Summary rendered as non‑offer content blocks inside checkout/post‑purchase surfaces.
- **theme.section** (confirmed) — **Pre‑purchase** widgets placed on **product page and cart page** via Shopify's theme editor / app blocks (beta): product upsell, recommendations, bundle, "frequently bought together", cart drawer upsell.
- **functions.discountRules** (inferred) — Upsell offers apply order/line discounts (% , fixed per unit, fixed per order) with a minimum‑order‑value gate; implemented as Shopify discounts/functions or draft‑order pricing rather than merchant‑managed codes.
- **functions.deliveryCustomization** (inferred) — "Shipping customizations" article: shipping/free‑shipping options exposed on Post‑Purchase, Thank‑You, and Order‑Status pages.
- **analytics.pixel** (confirmed) — Merchant‑configurable **pixel tracking** on the post‑purchase page (FB/GA/GA4); plus internal conversion/CTR/funnel analytics dashboard.
- **admin.block** (inferred) — Embedded admin app: Flow Builder, Editor, Dashboard, A/B test config, billing/plan management (the "where is the plan section" complaint implies an in‑admin billing surface).
- **flow.automation** (partial / inferred) — Not Shopify Flow. Its own rule engine ("Conditions") IS a lightweight automation/segmentation layer (see functional_model). Order side‑effects: auto‑**tag/note orders** generated by Upsell.com, and set order status to **"on hold" during reserve time** for post‑purchase upsells.

**How surfaces coordinate (confirmed):** The Flow Builder is the shared control plane. **Pages** (surfaces) contain **Conditions** (segmentation rules), which contain **Slots** (fixed on‑page locations placed once in the Shopify editor), which contain **Widgets** (groups), which contain **Blocks** (atoms). A Slot's physical location is pinned in the Shopify theme/checkout editor; Conditions then swap WHICH widget group fills that slot per customer. Priority is top‑to‑bottom: the first satisfied condition wins and evaluation stops. A **Default condition** (undeletable, pinned to bottom) is the catch‑all. Empty‑slot fall‑through is explicitly configurable ("continue to the next condition" vs "keep the slot empty"). Cross‑surface handoff: a customer accepting a post‑purchase one‑click upsell amends the SAME order (no re‑auth), and "previous offer" conditions chain upsell → downsell → cross‑sell across the funnel.

## functional_model
Concrete entities (current Flow Builder model):

- **Page** = { surface_id (Checkout | ProductPage | CartPage | ThankYou | OrderStatus | PostPurchase), conditions[] } — the surface being edited.
- **Condition** = { name, enabled(bool), logic(AND | OR), rules[], empty_slot_behavior(continue | keep_empty), priority(order_index), is_default(bool) } — segmentation rule; legacy name "trigger".
- **Rule** (within a condition) = { field, operator, value } where field ∈ { product‑in‑cart, product‑collection, cart/order‑value (>, <, =), customer‑segment (e.g. "purchased at least once" / new vs returning), customer info, previous‑offer‑outcome }. Combined via AND/OR.
- **Slot** = { number (auto, immutable — used to place it in Shopify editor), name (editable, internal), status(Live | Draft), widgets[] } — a container pinned to one physical location; up to **10 slots per condition**; unlimited widgets per slot.
- **Widget** = one of a fixed catalog (see settings_taxonomy): product‑upsell | product‑recommendations | bundle | banner | announcement‑bar | timer | birthday‑collector | custom‑form | order‑summary. Each widget = { type, enabled, ab_variants[], blocks[] }.
- **Block** = atomic UI element inside a widget: text | divider | image | product | product‑title | product‑details | product‑description | price‑&‑discount | accept‑button | decline‑button | variant‑picker | quantity‑selector | success/error‑banner | products | form‑fields.
- **Offer** (emergent) = { source_products, discount, quantity_rules, timer, accept/decline actions } expressed through the product/price‑&‑discount/accept‑button blocks. Accepting mutates the order/cart.
- **A/B test** = { widget, variants[], traffic_split, winning_metric } attached at the widget level.
- **Discount** = { type(percent | fixed_per_unit | fixed_per_order), amount, min_order_value } — applied to the offer, NOT a shareable code.
- **Recommendation source** = { strategy(shopify_recommendations{intent: related|complementary} | specific_products | collection{random?} | most_expensive_in_cart | cheapest_in_cart | third_party{Personalized Recommendations|Recomatic|Wiser|Rebuy|Recomatic}), exclusion_tags[], hide_cart_products(bool) }.
- **Order side‑effects** = { tag, note, status_override("on hold" during reserve/reserve_time) }.

Relationships: Page 1→N Condition 1→N Slot(≤10) 1→N Widget 1→N Block. Condition selects Widget‑group per Slot per customer. Offer/Discount/Recommendation are widget/block configuration, not standalone records the merchant CRUD's.

## mapping_note
Maps onto our RecipeSpec vocabulary only at the leaf level; the app as a whole is a **multi‑surface blueprint + rule engine + stateful backend**, which is exactly where it exceeds a single module.

- A single **widget** (e.g. one Product Upsell offer block on the thank‑you page, with a fixed product, one discount, one timer) ≈ one RecipeSpec module: `postPurchase.offer` / `checkout.upsell` / `theme.section` with content + style + a discount rule. This part is in‑scope.
- Where it **EXCEEDS a single‑module recipe**:
  1. **Cross‑surface blueprint with shared state**: Pages → Conditions → Slots → Widgets is one coordinated funnel spanning product, cart, checkout, post‑purchase, thank‑you, and order‑status — a composeBlueprint of multiple extension types that hand off (accept → amend same order → chained downsell/cross‑sell), not one module.
  2. **A real rule/segmentation engine**: prioritized, AND/OR, top‑to‑bottom‑evaluated Conditions with a default catch‑all, empty‑slot fall‑through, and segment/collection/cart‑value/previous‑offer targeting. This is a runtime evaluator with per‑customer branching, not static module config.
  3. **A persistent external data store + background side‑effects**: flow definitions, A/B test results, analytics/attribution, captured birthday/form data, usage/billing counters, plus order mutations (add line item, apply discount, tag/note order, set "on hold" during a reserve window). Requires a DB and job/timer machinery a single recipe can't own.
  4. **External side‑effects & integrations**: applies Shopify discounts/Functions, fires FB/GA/GA4 pixels, and syncs to Klaviyo/Omnisend/Loox/Recharge/rec‑engines — outbound effects beyond the module boundary.

Net: recreate the *vocabulary of one widget/offer* inside RecipeSpec; but the ReConvert/Upsell.com product is a **blueprint of many surface modules bound by a condition engine and a stateful backend with background jobs and external side‑effects**.
